## 一、结论速览

这个项目的**架构分层是清晰的**（事件记录 → 蒸馏 → 晋升候选，人审后再改 SKILL），测试覆盖也覆盖了主干路径。但它存在一个**核心可用性断层**和若干**正确性 / 设计缺陷**，导致"从真实使用中自动学习"这条主链路在实践中大概率不会真正触发：

1. **纠正检测只认中文**，而 README 明确承诺识别英文纠正 —— 文档与代码直接矛盾（已实测）。
2. **蒸馏聚合键用自由文本精确匹配**，几乎不可能聚成同一 pattern，`distill` 在真实数据上基本永远产不出候选。
3. **事件 ID 按秒生成会碰撞**（已实测），破坏可追溯性。
4. **CLI 缺少 `record` / `list` / `status` 命令**，用户想手动记录或查看学习结果只能去调内部脚本，与 README「脚本仅供调试」的定位冲突。

下面详述。

---

## 二、严重（Critical）—— 影响核心功能能否成立

### C1. 纠正检测与文档承诺不一致：英文纠正完全无法识别
- **证据（实测）**：
  ```
  "thats wrong"            -> false
  "no, not that"           -> false
  "you should have used X" -> false
  ```
  而 `README.md` 明确写道：
  > a correction like "wrong", "not that", or "you should have..."
- **根因**：`scripts/hook-utils.js` 中 `CORRECTION_PREFIXES` 与 `looksLikeCorrection` 的正则**全部是中文**（`不对/不是/应该/不要…`）。
- **影响**：面向 Claude Code / Codex 的英文用户，自动学习链路的入口（UserPromptSubmit 纠正判定）**永不触发**，整套 hook 机制对英文用户形同虚设。README 是英文主文档，问题尤其突出。
- **建议**：要么补齐英文纠正词表/语义判定，要么把 README 承诺改为「仅中文」。二者取一，消除文档-代码矛盾。

### C2. 蒸馏聚合键是"自由文本精确匹配"，候选几乎永远产不出
- **根因**：`distill-patterns.js` 的 `buildPatternKey` = `target_scope :: target_name :: proposed_lesson`。而 hook 路径下 `proposed_lesson` 来自 `inferLesson(prompt)`，本质是**用户纠正原话去掉前缀**。
- **影响**：两条含义相同但措辞不同的纠正（"应该先看文档" / "要先读文档再动手"）会生成**不同的 key**，无法聚合。于是 `evidence_count >= 3` 这个晋升门槛在真实数据里**几乎不可能达到**，`distill` 长期只会输出 "No promoted candidates yet."。
- **佐证**：现有测试之所以能"蒸馏出候选"，是因为测试数据人为地让多条事件 `proposed_lesson` **逐字相同**——这在真实使用中不成立。
- **建议**：聚合应基于**归一化/语义相似**（如去停用词 + 相似度聚类，或按 `target_name + signal_type` 粗聚合后再看 lesson），而非整句精确匹配。

### C3. `distinct_tasks` 指标在 hook 路径下被稀释，晋升门槛失真
- **根因**：`createPattern` 用 `task_id || task_summary` 计 distinct tasks；hook 记录时 `task_summary` = 整条用户 prompt、`task_id` = `turn_id`。每次纠正 prompt 几乎都不同 → **每条证据都被算作一个"不同任务"**。
- **影响**：`distinct_tasks >= 2` 这个"跨任务复现"的约束基本失效，无法真正区分"同一任务反复纠正"与"多任务共性问题"。与 C2 叠加，使阈值语义整体不可信。
- **建议**：为一次真实任务分配稳定的 `task_id`（会话/任务级别），而非用逐条 prompt 充当任务标识。

## 三、高（High）—— 正确性与可用性缺陷

### H1. 事件 ID 按秒生成，同秒必然碰撞（已实测）
- **证据**：`compactDateStamp` 取 ISO 前 14 位（`YYYYMMDDHHMMSS`）：
  ```
  2026-07-03T10:00:00.111Z 与 ...00.999Z -> 同为 evt-20260703100000
  ```
- **影响**：同一秒内的多条事件 `id` 相同 → `events.jsonl` 出现重复 ID、`evidence_refs` 指向不唯一、人工追溯困难。
- **建议**：ID 追加毫秒/随机后缀或自增序号，保证唯一。

### H2. CLI 缺少一等公民命令：无法记录 / 查看学习结果
- **现象**：`bin/cli.js` 仅有 `install / doctor / uninstall / distill`。**没有** `record`（手动记一条）、`list` / `show`（查看事件或候选）、`status`（概览）。
- **矛盾**：README 说 CLI 是"团队标准路径"、内部脚本"供测试/调试"，但用户要手动记一条学习或查看候选**只能**去 `node scripts/log-learning.js …`，正好落在被贬为"调试用"的入口上。
- **影响**：日常最高频的两个动作（记录、查看）没有稳定的产品化入口，产品闭环缺口明显。
- **建议**：把 `record`（包装 `record-from-context`）与 `list/status`（读取 `.learnings/`）提升为 CLI 命令。

### H3. 纠正判定误报率高（正常提问被当成纠正）
- **证据（实测）**：
  ```
  "应该怎么做才能不要报错"   -> true（其实是提问）
  "请告诉我不要用哪个库"     -> true（其实是提问）
  ```
- **根因**：`不要 / 别 / 应该` 等高频词只要出现（甚至只是前缀）就判为纠正。
- **影响**：把普通提问误记为"用户纠正"，污染 `.learnings/`，拉低后续蒸馏质量。与 C1（漏报英文）形成"该报的不报、不该报的乱报"。
- **建议**：收紧规则（要求紧跟被追踪 skill 的上下文、结合否定+祈使结构），或引入轻量语义判定。

### H4. 运行时状态是**单一全局文件**，多会话并发会串味
- **根因**：`runtime-state.js` 把"最近使用的 skill / observed"写在**唯一**的 `.learnings/runtime/last-skill-context.json`。
- **影响**：同一机器上并行开两个 Claude/Codex 会话时，A 会话的 skill 上下文会被 B 覆盖；随后 A 的纠正会被**错误归属**到 B 的 skill。归属本就粗糙（见 H5），并发下更不可靠。
- **建议**：按会话/`turn_id`/项目路径隔离状态，或在事件里校验 skill 与会话一致性。

### H5. 纠正归属过于粗糙：只记"最近一次 slash skill"
- **现象**：`handlePromptPayload` 只认最近一次被追踪的 slash 命令。若用户 `/skillA` 后又进行了多轮普通对话再纠正，纠正仍会被归到 `skillA`，即便早已跑题。
- **影响**：证据的"归属 skill"可信度低，直接影响后续按 `target_name` 的聚合与晋升。
- **建议**：为追踪加入时效/轮次衰减，或在超过 N 轮后不再自动归因。

## 四、中（Medium）—— 健壮性、可移植性、隐私

### M1. Windows 事实上不受支持
- **根因**：安装写入的 hook `command` 是**裸 `.js` 路径**（依赖 shebang + 可执行位），且 `path.join` 在 Windows 会产出反斜杠路径写进 JSON。
- **影响**：Windows 上 hook 命令基本无法执行；`package.json` 只声明 `node>=18`，未声明 OS，用户会踩坑。
- **建议**：hook 命令改为 `node <abs-path>` 形式；或在文档/`os` 字段中明确平台支持范围。

### M2. npm 安装后脚本可执行位可能丢失，hook 静默失效
- **根因**：hook 依赖脚本可直接执行；`install-shared.copySkillTree` 用 `fs.cpSync`（本地拷贝会保留 mode），但**经 npm 发布再安装**的文件不保证保留可执行位，而这些脚本又不在 `package.json` 的 `bin` 里。
- **影响**：`doctor` 只检查"命令是否写进配置"，**不检查脚本能否真正被执行**，因此这种失效是静默的。
- **建议**：hook 统一用 `node <path>` 调用（绕开可执行位问题），并让 `doctor` 增加"脚本可执行/可被 node 运行"的探测。

### M3. 自动捕获 prompt / assistant 全文落盘，隐私提示与实际行为不匹配
- **现象**：Stop hook 把**整段** assistant 输出写入 `observed`；纠正事件把用户 prompt 原文写入 `.learnings/`。README 有隐私声明，但**每条事件是静默自动记录**、无逐条同意、无脱敏、无大小上限。
- **影响**：可能把密钥、内部信息等写进本地明文文件；`observed` 无截断，单条可非常大。
- **建议**：对 `observed`/`feedback` 做长度截断与可选脱敏；或提供"仅摘要不存原文"的开关。

### M4. `doctor` 的可写检查有副作用
- **根因**：`dirWritable` 通过 `fs.mkdirSync(...)` 真正**创建** `.learnings` 目录来判断可写。
- **影响**：一个"只读诊断"命令会在磁盘上留下目录，语义不纯。
- **建议**：用 `fs.accessSync(dirname, W_OK)` 之类的非破坏性探测。

### M5. 无自动蒸馏触发，学习闭环靠人记得手动跑 `distill`
- **现象**：没有任何 hook / 定时机制触发蒸馏，候选只有在用户主动 `npx … distill` 时才刷新。
- **影响**：即便事件积累到位，晋升候选也常年不更新，"持续改进"退化为"偶尔手动汇总"。
- **建议**：可在 Stop hook 后按事件计数阈值触发轻量蒸馏，或在 `doctor` 中提示"有 N 条新事件待蒸馏"。

### M6. 记录无幂等 / 去重，重复纠正会重复入库
- **现象**：`log-learning` 每次调用都 append，无内容/时间去重。
- **影响**：同一纠正被 hook 多次触发（或用户重复表达）会产生多条几乎相同的证据，虚增 `evidence_count`。
- **建议**：按 (skill, lesson 归一化, 近窗口) 去重或合并计数。

## 五、低（Low）—— 打磨项

- **L1. `readJsonStdin` 同步读 fd 0**：无 stdin 时 `fs.readFileSync(0)` 可能抛错/阻塞，hook 环境一般有输入，但缺少防御。
- **L2. `inferLesson` 产出的"lesson"其实是用户抱怨原话去前缀**，并非"可复用指令"，与 SKILL.md「expressible as a reusable instruction」的要求有落差。
- **L3. 事件 schema / 状态文件无版本号**，未来字段演进缺乏迁移位。
- **L4. 三处 `parseArgs` 重复实现**（cli / log-learning / record-from-context / distill / install / uninstall 各写一份），风格略有差异（如布尔标志处理不一致），可抽公共解析。
- **L5. Codex hook 配置格式为推测**：`HOOKS_INSTALL.md` 自述 codex 也可能用 `config.toml`，但安装器只写 `hooks.json` 且沿用 Claude 的 group/hooks 结构，未见与真实 Codex 规范对齐的验证。
- **L6. 测试盲区**：英文纠正、ID 碰撞、`record-from-context --context-json` 文件路径、doctor 未安装分支、并发状态覆盖等均无用例；现有蒸馏测试因数据"逐字相同"而掩盖了 C2。
