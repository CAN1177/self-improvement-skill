# self-improvement-skill 项目不足点分析

生成时间：2026-07-03 14:14:48

说明：本分析未读取 `todo.md` / `TODO.md` 的内容，只基于项目当前可见的 README、SKILL、CLI、脚本、hook 实现和测试用例。

## 一句话判断

这个项目已经有清晰的最小闭环：安装 skill、写入 Claude/Codex hooks、捕获最近一次 slash skill、在用户纠正时记录证据、再按阈值蒸馏候选规则。当前不足主要不在“能不能跑”，而在真实使用环境里的鲁棒性、证据质量、配置兼容性、并发安全和候选规则质量。

## 当前做得比较稳的部分

- 项目边界清晰：只记录证据和候选规则，不直接改写其它 `SKILL.md`。
- CLI 路径简单：`install`、`doctor`、`uninstall`、`distill` 覆盖了基础生命周期。
- 测试覆盖了主要 happy path：安装、卸载、hook 记录、事件落盘、模式蒸馏、CLI 参数解析。
- 安装时会排除 `.learnings`，避免把本地学习数据复制到全局安装目录。
- 中英文 README 基本同步，适合团队内外使用。

## 高优先级不足

### 1. hook 状态只有一个全局槽，容易串上下文

相关文件：

- `scripts/runtime-state.js`
- `scripts/hook-utils.js`

当前只把最近一次 slash skill 存在 `.learnings/runtime/last-skill-context.json`。如果用户在多个会话、多个终端、多个并发任务里交替使用 skill，后一次 slash skill 会覆盖前一次上下文。随后用户纠正时，事件可能被记录到错误的 skill 上。

影响：

- 学习证据会污染目标 skill。
- 错误证据进入 distill 后，会生成错误候选规则。
- 用户越依赖自动 hook，误归因风险越高。

建议：

- 按 `session_id` / `conversation_id` / `turn_id` 维护状态，而不是单文件单槽。
- 如果宿主 payload 没有稳定会话 ID，至少把最近 N 个上下文按时间保留，并在纠正时用 turn 距离或时间窗口匹配。
- 对缺少会话 ID 的平台，在事件里标记 `confidence: low`。

### 2. 事件 ID 只精确到秒，短时间多次记录会碰撞

相关文件：

- `scripts/log-learning.js`

事件 ID 当前为 `evt-${YYYYMMDDHHMMSS}`。同一秒内多次记录会生成相同 ID。JSONL 仍会追加成功，但后续 evidence refs、Markdown 引用和候选规则会出现重复 ID，审计价值下降。

建议：

- ID 加入毫秒、随机后缀或递增序号，例如 `evt-20260703141448-7f3a`。
- 测试覆盖同一秒两条事件的唯一性。

### 3. 写文件没有并发保护，hooks 并发触发时可能丢数据或写坏 JSONL

相关文件：

- `scripts/log-learning.js`
- `scripts/runtime-state.js`
- `scripts/distill-patterns.js`

当前写入依赖 `appendFileSync` / `writeFileSync`，没有锁、临时文件 rename、重试或冲突检测。单用户低频场景问题不大，但 hook 运行在宿主生命周期里，多个会话并发触发时会有风险。

影响：

- `last-skill-context.json` 被覆盖。
- `patterns.json` 和 `promotion_candidates.md` 可能被并发 distill 覆盖。
- JSONL 追加大概率还行，但没有完整性校验。

建议：

- 对状态文件和 distill 输出使用原子写：写临时文件后 rename。
- 对 `events.jsonl` 加轻量文件锁，或把每个事件先写入独立 event 文件，再异步归并。
- doctor 增加 `.learnings` 文件完整性检查。

### 4. Codex hook 配置实现和文档描述不完全一致

相关文件：

- `HOOKS_INSTALL.md`
- `scripts/install-codex-hooks.js`
- `scripts/install-paths.js`

文档说 Codex hooks 可以在 `~/.codex/hooks.json` 或 `~/.codex/config.toml`，但实现只读写 `~/.codex/hooks.json`。如果用户实际环境使用 `config.toml`，安装结果可能显示正常，但真实 Codex 不一定加载这些 hooks。

建议：

- 明确支持范围：要么文档只承诺 `hooks.json`，要么实现检测/写入 `config.toml`。
- doctor 应检查当前 Codex 实际加载路径，而不是只检查目标文件存在。
- 集成测试增加 Codex 配置路径差异场景。

### 5. hook payload 解析过于猜字段，缺少平台契约校验

相关文件：

- `scripts/hook-utils.js`
- `scripts/claude-hook-*.js`
- `scripts/codex-hook-*.js`

`extractPrompt` 和 `extractAssistantMessage` 通过多个可能字段猜测值：`prompt`、`user_prompt`、`message`、`input`、`text`、`response`、`output_text` 等。这样兼容性好，但没有记录到底命中了哪个字段，也没有平台版本契约。

影响：

- 某些 payload 中 `message` 可能不是用户 prompt 或助手输出，导致误记录。
- 字段缺失时只静默 skip，用户不知道 hook 没工作。

建议：

- 为 Claude 和 Codex 分别维护 payload schema / fixture。
- hook 执行时在 debug 模式输出命中字段和 skip 原因。
- doctor 增加一次模拟 hook payload 自检。

## 中优先级不足

### 6. 自动判断“纠正”的规则太粗

相关文件：

- `scripts/hook-utils.js`

当前通过中文前缀和少量正则判断纠正，如“不对”“不是”“应该”“改成”。这能覆盖常见中文纠正，但也会漏掉很多真实表达，例如英文反馈、委婉反馈、只说“这里方向错了”、或者多句里后半句才是纠正。也可能误把普通需求变更当成纠正。

建议：

- 把 `looksLikeCorrection` 输出改成 `{ matched, reason, confidence }`。
- 支持可配置关键词。
- 引入“记录前最低上下文完整度”，例如必须有 `used_skill` 且 `observed` 非空，低置信度只进入待确认队列。

### 7. lesson 生成只是简单去前缀，质量不稳定

相关文件：

- `scripts/hook-utils.js`

`inferLesson` 只是移除“不对/不是/应该”等前缀。用户原话经常是情绪化、场景化、缺主语或缺动作的，直接作为长期候选 lesson 会让 distill 结果质量不稳定。

建议：

- 自动记录原始 `feedback`，但 `lesson` 应标为 `raw_inferred_lesson`。
- 后续 distill 时再生成更规范的 `candidate_rule`。
- 候选规则至少包含动作、适用条件、例外边界。

### 8. distill 聚类过度依赖完全相同 lesson

相关文件：

- `scripts/distill-patterns.js`

模式 key 是 `target_scope + target_name + proposed_lesson` 的规范化字符串。只要用户用不同措辞表达同一问题，就会分裂成多个 pattern，导致永远达不到 `evidence_count >= 3`。

建议：

- 先保留当前确定性聚类，再增加人工可读的“相似候选”列表。
- 允许按 `target_name + signal_type + normalized task theme` 辅助聚合。
- 对中文同义表达做更强的归一化，至少处理标点、常见停用词和“先/优先/首先”等近义表达。

### 9. Markdown 日志没有转义，多行用户输入会破坏结构

相关文件：

- `scripts/log-learning.js`

`appendMarkdown` 直接把用户输入拼到列表项里。若 `feedback`、`lesson`、`what_happened` 包含换行、Markdown 标题、代码块、列表或表格，会导致日志结构混乱，甚至让后续人工审查误读。

建议：

- 多行字段使用 fenced code block。
- 单行字段做换行替换或缩进。
- 测试覆盖多行反馈、代码片段、Markdown 标题等输入。

### 10. 安装会重置目标 skill 目录，可能误删用户手工放入的文件

相关文件：

- `scripts/install-shared.js`

`copySkillTree` 会 `resetDir(targetDir)`，然后复制当前包内容。虽然 `.learnings` 会排除并重新创建 runtime 目录，但如果用户在已安装目录下手工放了其它本地文件，会被删除。

建议：

- 安装目录里只管理明确文件清单，避免整目录删除。
- 或引入 `.managed-by-self-improvement-skill` 清单，只删除上次安装过的文件。
- 至少在 README 里明确“安装会重建 skill 目录，除 `.learnings` 外不保留其它文件”。

### 11. 卸载 hook 的识别方式只看 basename，可能误删其它同名脚本

相关文件：

- `scripts/install-shared.js`
- `scripts/uninstall-team-skill.js`

`removeManagedCommandHooks` 通过 `path.basename(hook.command)` 判断是否受管理。只要其它配置里也有同名脚本，就会被移除。

建议：

- hook 写入时添加稳定 marker，例如 `SELF_IMPROVEMENT_MANAGED` 环境变量或 wrapper 注释字段。
- 卸载时同时检查路径是否位于目标 skill 安装目录。
- 测试覆盖“同名但不同来源脚本不能被删”的场景。

### 12. doctor 的诊断深度不够

相关文件：

- `scripts/doctor.js`

doctor 当前检查 skill 文件、config 文件、hook 命令是否存在于 JSON、`.learnings` 是否可写。它不能证明 hooks 真的可执行、shebang 有权限、Node 版本满足、Codex/Claude 会加载该配置、JSON 是否有非预期结构。

建议：

- 检查 `process.version` 是否满足 package engines。
- 检查 hook 脚本是否存在、可执行或至少可通过 `node script < sample.json` 运行。
- 检查 `.learnings/events.jsonl` 是否可追加一条 dry-run 记录。
- 输出可操作修复建议，而不是只有 yes/no。

## 低优先级不足

### 13. CLI 参数体验偏简陋

相关文件：

- `bin/cli.js`
- 各 `scripts/*.js`

当前解析器不支持 `--target=codex`，也没有统一 help、版本号、错误码分类、dry-run。内部脚本和主 CLI 的布尔参数行为还不完全一致，有些缺值会变成 `"true"`，有些会报错。

建议：

- 统一参数解析规则。
- 支持 `--version`、`--dry-run`。
- 对安装/卸载增加变更预览。

### 14. 发布与质量门禁缺少自动化

当前 `package.json` 只有 `test`，没有 lint、format、typecheck、prepack、release 检查。作为 npm 包，发布前最好自动验证 files 清单、bin 可执行、README 中命令可用。

建议：

- 增加 `npm pack --dry-run` 检查。
- 增加 smoke test：安装到临时 HOME 后运行 CLI。
- 增加 changelog 或 release checklist。

### 15. 隐私策略只有文档提醒，缺少工具级防护

相关文件：

- README
- `scripts/log-learning.js`

项目正确提醒 `.learnings` 可能包含敏感数据，但工具没有提供脱敏、查看摘要、清理或导出前审查能力。

建议：

- 增加 `redact` 或 `export --redacted` 命令。
- 默认截断超长 assistant output。
- 对 API key、token、邮箱、手机号等常见敏感模式做可选脱敏。

### 16. 缺少真实宿主的端到端验证

当前测试大多直接调用函数，并手工构造 payload。它能保证内部逻辑，但不能保证 Claude Code / Codex 的真实 hook payload、加载路径、工作目录、stdin 行为和权限模型都匹配。

建议：

- 保存真实 Claude/Codex hook payload fixture。
- 用 fixture 跑回归测试。
- 维护一份手工验收清单：安装、触发 slash skill、Stop 捕获、纠正记录、distill 输出、卸载。

## 建议优先级路线

### 第一阶段：先守住证据正确性

1. 事件 ID 唯一化。
2. 状态文件按会话/turn 隔离。
3. Markdown 日志字段安全渲染。
4. 卸载时避免 basename 误删。

### 第二阶段：提升真实环境可用性

1. 明确并修复 Codex 配置路径支持。
2. doctor 增加 hook 可执行检查和模拟 payload 检查。
3. 为 Claude/Codex 分别补真实 payload fixture。
4. 安装/卸载增加 dry-run。

### 第三阶段：提升学习质量

1. correction 检测输出置信度和原因。
2. lesson 与 raw feedback 分离。
3. distill 支持相似表达聚合。
4. 候选规则输出适用条件、动作、反例和证据引用。
